# 1. AroundCities Is No Longer Just An Event Site

The biggest shift.

Originally:

```txt
AroundCities = Event Listing Website
```

Now:

```txt
AroundCities = A Living Digital Layer Of Kuching
```

Potential content:

* Events
* Food
* Atmosphere photos
* Event photos
* Positive messages
* Future local creators

The goal became:

> Make Kuching feel alive.

---

# 2. Homepage Should Be Card-Based

This was probably the most important architecture decision.

Instead of:

```txt
Events
Photos
Greetings
Food
```

all being separate systems,

everything becomes:

```txt
Cards
```

Homepage displays cards.

Cards can represent:

```txt
event
photo
food
registration
positive_message
```

This greatly simplifies homepage rendering.

# 4. User Visit Tracking

Important conclusion:

Don't build:

```txt
user_visits
```

yet.

Instead:

### Use Vercel Analytics

For:

* visitors
* page views
* peak traffic times
* devices
* countries

---

### Use card_stats

For:

* card popularity
* hourly trends
* content performance

Example:

```txt
Food Festival

7pm  = 80 views
8pm  = 200 views
9pm  = 140 views
```

---

# 5. Homepage Freshness Strategy

Problem:

Users revisit homepage.

You don't want them to always see:

```txt
Greeting
Summary
Same content
```

Solution discussed:

Use localStorage.

Example:

```json
{
  "card_123": "2026-05-31T09:00"
}
```

Then:

* Seen recently → lower priority
* Unseen → higher priority

No database required.

## Event Photos

These are tied to events.

# 9. Mobile App Strategy

Important conclusion:

Don't build native apps now.

---

Current path:

```txt
Responsive Website
    ↓
PWA
    ↓
Wrapped App
```

---

Why?

You correctly pointed out:

> Users trust App Store more than manual PWA installation.

This is true.

Long term:

```txt
Web
+
Capacitor wrapper
```

# 12. Content Policy Discovery

The TVS Gawai poster exposed another important realization.

Not everything from Facebook should become an event.

You need editorial judgment.

Question:

> Would a Kuching resident appreciate seeing this?

If yes:

Feature it.

If not:

Ignore it.

This is probably going to become more important than database design.

---

# Biggest Takeaway

The strongest idea that emerged during your vacation was:

> AroundCities should not feel like an event database.

It should feel like:

> A local person paying attention to what is happening in Kuching and sharing it through cards.

Everything we discussed—photos, event moments, greetings, food, homepage behavior, community photographers—supports that vision.

If I were sitting at your workstation today, I would start with only three things:

1. Finalize the card-centric database structure.
2. Build the homepage card feed.
3. Build event creation and photo creation workflows.

Everything else can evolve from there.
